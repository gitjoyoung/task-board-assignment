# Task Board — 느리고 가끔 실패하는 서버 위의 칸반 보드

> 과제 원문(요구사항 전문)은 [docs/assignment.md](docs/assignment.md)에 그대로 보존했어요.

- **배포**: https://gitjoyoung.github.io/task-board-assignment/
- **설계 결정과 근거**: [DECISIONS.md](DECISIONS.md) · **AI 활용 기록**: [AI_USAGE.md](AI_USAGE.md)

## 과제를 이렇게 이해했어요

칸반 보드는 벽에 붙인 포스트잇의 은유예요. 사용자가 기대하는 건 두 가지라고 봤어요 — 포스트잇을 옮기는 것만큼 **즉각적으로 반응**할 것, 그리고 옮긴 것은 **반드시 거기 있을 것**(화면이 거짓말하지 않을 것). 그런데 이 과제의 서버는 느리고(200~800ms) 가끔 실패해서(15%), 이 둘이 서로 충돌해요. 즉각적이려면 서버보다 먼저 보여줘야 하고, 먼저 보여주면 거짓이 될 위험이 생기니까요. 구현한 모든 것(낙관적 반영, 롤백, 실패 큐, 카드별 직렬화, 오프라인 선발행, 다중 탭 방송)은 이 충돌의 중재안이에요.

## 실행 방법

```bash
npm install         # postinstall 에서 public/mockServiceWorker.js 자동 생성
npm run dev         # 개발 서버 (화면 우하단에 DEV 패널이 떠요)
npm test            # 유닛·통합 테스트 72개
npm run test:coverage
npm run build       # 타입체크 + 프로덕션 빌드
```

Node 18 이상을 권장해요.

## 구현 기능

### Priority 1 — 전부 구현했어요

| 요구사항 | 구현 | 확인 방법 |
|---|---|---|
| 로드 상태 처리 | 로딩 / 에러(재시도 버튼) / 빈 상태 3분기, 초기 로드는 조용한 재시도 2회 | DEV 패널에서 읽기 실패율 1로 올리고 새로고침 |
| 낙관적 업데이트 + 롤백 | 이동·수정·삭제·생성 전부 즉시 반영, 실패 확정(~2초) 시 그 카드만 서버 확정 상태로 복원 | 쓰기 실패율 1로 올리고 드래그 |
| 경쟁 상태 | 카드별 직렬화 + version 승계 + 연타 병합 + 늦은 응답 무시 | 같은 카드 빠르게 연속 이동 후 새로고침 |
| 5,000개 성능 | 고정 높이 가상화 — DOM 카드 약 54개, 스크롤 프레임 평균 4.1ms | 스크롤·드래그·검색 |
| 태스크 CRUD | 네이티브 dialog, 추가(제목·우선순위 필수)/수정/삭제(2단계 확인) | 컬럼 헤더 +, 카드 클릭 |
| 핵심 로직 테스트 | 72개 (동기화 정책 모듈에 가장 두껍게), 라인 커버리지 93% | `npm test` |

### Priority 2 — 4개 구현했어요

- **409 충돌 처리**: 서버 최신 상태 채택 + 통보. 카드별 직렬화 덕분에 자기 자신과의 409는 아예 발생하지 않아요
- **실패 재시도**: 자동 1회(0.3초 백오프) 후 실패 큐 누적. 알림에 무엇을 하다 실패했는지 항목별로 표시되고, 일괄 재시도를 누르면 알림이 유지된 채 성공한 항목부터 빠져요
- **검색 디바운싱 + 우선순위 필터**: 입력은 즉시, 필터 계산만 250ms 지연
- **다중 탭 동기화**: 서버가 확정한 변경만 BroadcastChannel로 방송, 받는 쪽은 version이 더 새것일 때만 반영 (추가 요청 0건)

### 부가: 오프라인 처리와 DEV 패널

- **네트워크 단절**: 요청 전에 연결을 확인하고, 오프라인이면 재시도 없이 즉시 알려요. 이동·수정은 화면에 먼저 반영해 두고 연결이 돌아오면 자동 재전송해요
- **DEV 패널** (개발 모드 전용, 우하단): 쓰기/읽기 실패율, 지연 범위, 네트워크 끊김을 화면에서 바로 조절할 수 있어요. 위 표의 "확인 방법"이 전부 이 패널로 재현돼요

## 미구현과 사유

- **키보드 접근성** — P1 완성도에 시간을 집중하느라 뺐어요. 네이티브 dialog의 기본 포커스 트랩까지만 확보된 상태예요
- **태그·다중 필터** — 같은 이유로 우선순위 단일 필터까지만 구현했어요
- **실패 큐 영속화, 백오프 지터, 멱등성 키** — 몰라서가 아니라 범위 판단으로 뺐고, 각각 언제 다시 넣어야 하는지를 [DECISIONS.md](DECISIONS.md) 6번에 적어 뒀어요
