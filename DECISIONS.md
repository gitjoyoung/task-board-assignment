# 설계 결정 (DECISIONS)

> 이 문서는 평가에서 **20%**를 차지합니다. "무엇을 했는지"보다 **"왜 그렇게 했는지"**를 봅니다.
> 아래 항목을 채워 주세요. 코드로 답이 되는 부분은 파일 경로를 함께 적어도 좋습니다.

## 0. 설계 축

이 과제의 본질을 "서버는 느리고(200~800ms) 가끔 실패하는데(15%), 사용자에게는 즉각적이고 정직한 UI를 보여주는 것"으로 정의했다. 아래 모든 결정은 이 축(즉각성 ↔ 정합성) 위에서의 트레이드오프다.

## 1. 상태 구조
- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?
- 이렇게 잡은 이유와, 고려했다가 버린 대안은?

**결정: TanStack Query 도입. 서버 상태는 `['tasks']` 쿼리 캐시 하나로 관리하고, 낙관적 상태는 같은 캐시에 즉시 반영한다.**

- **invalidate → 재조회 패턴을 버렸다.** 이 API는 단건 조회가 없고 `GET /api/tasks`가 5,000개 전체를 반환한다. 교과서대로 뮤테이션마다 invalidate하면 카드 한 장 옮길 때마다 5,000개를 200~800ms 걸려 다시 받고, 그 GET도 가끔 500이 난다. 대신 **뮤테이션 응답(서버가 최신 task를 돌려줌)으로 캐시를 직접 갱신**한다. GET은 최초 1회 + 사용자가 명시적으로 재시도할 때만.
- 그래서 `staleTime: Infinity`, `refetchOnWindowFocus: false`. 자동 재조회가 낙관적 상태를 덮어쓰는 사고도 함께 막힌다.
- 초기 로드는 `retry: 2` — 명세상 GET 실패는 "드물게 500"이므로 조용한 재시도로 흡수하고, 그래도 실패하면 재시도 버튼이 있는 에러 UI를 보여준다 (`src/Board.tsx`).
- **버린 대안: `useReducer` 직접 구현.** 의존성 0이라는 장점이 있지만, 요청 취소·중복 요청 방지·스냅샷 롤백을 전부 손으로 짜야 하고 그 코드는 아무도 검증해 준 적이 없다. Query는 이 패턴이 공식 문서에 정형화되어 있다. 단, 같은 카드 연속 이동의 경쟁 상태는 Query가 자동으로 해결해주지 않으므로 3번에서 직접 보완한다.
- 부수 효과: React 18 StrictMode의 effect 이중 실행으로 초기 GET이 2번 나가던 문제도 Query의 요청 중복 제거로 해소.

## 2. 낙관적 업데이트 & 롤백
- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?
- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?

**결정: 스냅샷(baseline) 복원. 롤백돼도 변경 의도는 실패 큐에 보존되어 사용자가 일괄 재시도할 수 있다.** (`src/lib/taskMover.ts`)

- 모든 뮤테이션은 캐시에 즉시 반영 → 백그라운드 전송. 일시 오류는 백오프(0.3/0.6/1.2s)로
  3회 자동 재시도해 조용히 흡수하고(15% 실패율 기준 잔존 ~0.05%), 소진 시에만 롤백 + 알림.
- 롤백 목적지는 **baseline** — 그 카드를 두고 서버가 마지막으로 확정해 준 상태. 연속 이동 중
  두 번째가 실패하면 "처음"이 아니라 "첫 이동이 확정된 상태"로 돌아간다 (테스트로 고정).
- **서버 재조회를 버린 이유**: GET 이 5,000개 통짜라 롤백 한 번에 200~800ms + 실패 가능성을
  지불해야 하고, 재조회 응답이 다른 카드들의 진행 중인 낙관 상태까지 덮어쓴다. 스냅샷 복원은
  실패한 카드만 정확히 되돌린다.
- 알림은 [N건 재시도] [요청 취소] 2-선택지이며 어떤 작업(이동/수정/생성/삭제: 카드 제목)이
  실패했는지 항목별로 명시한다. 상세: docs/policy-decisions.md PD-2~PD-4.

## 3. 경쟁 상태(race condition)
- 같은 카드를 빠르게 연속 이동할 때 최종 상태가 어긋나지 않도록 무엇을 했나요?
  (요청 취소 / 버전·순번 비교 / last-write-wins 등)

**결정: 카드별 직렬화 큐 + version 승계 + 연타 병합.** (`src/lib/taskMover.ts`)

- **카드당 동시 요청 1개.** 진행 중에 들어온 변경은 `nextPatch` 로 병합돼 이전 응답이 도착한
  뒤 그 응답의 version 으로 이어서 전송된다. 병렬로 쐈다면 두 번째부터 전부 409였을 것 —
  자기 자신과의 충돌을 원천에서 막은 게 이 설계의 핵심이다.
- **연타 합치기**: A→B→C→D 로 빠르게 옮기면 중간 상태(B, C)는 서버로 보내지 않고 최종
  목표만 전송한다 (요청 수 절약 + 중간 상태가 서버에 남는 일 없음).
- **늦게 도착한 응답이 화면을 덮지 않는다**: 첫 요청의 응답이 왔을 때 더 새로운 낙관 상태가
  있으면 status 는 유지하고 서버 메타(version 등)만 채택한다. Figma 가 flicker 방지에 쓰는
  "미확인 로컬 변경 우선" 원리와 동일 (docs/enterprise-case-studies.md 2절).
- 같은 컬럼 드롭(no-op)은 요청 자체를 보내지 않는다 — 불필요한 PATCH 와 15% 확률의 가짜
  실패 알림 방지.
- 삭제만 예외적으로 직렬화하지 않는다: DELETE 는 version 무관 + 멱등이라 병렬 전송이 안전
  하고, 삭제 후 도착하는 수정 응답이 카드를 되살리지 않도록 별도 가드를 뒀다 (PD-9).
- 위 동작 전부 유닛 테스트로 고정 (`taskMover.test.ts` — 직렬화/version 승계/늦은 응답/
  연타 합치기/체인 중간 실패, 총 50개 중 26개가 이 모듈).

## 4. 대량 데이터 성능 (5,000개)
- 검색/필터/드래그를 부드럽게 만들기 위해 무엇을 선택했고 **무엇을 포기**했나요?
- 가능하면 측정 근거(예: 최적화 전/후 체감·프로파일)를 남겨 주세요.

**결정: @tanstack/react-virtual 고정 높이 가상화 + 디바운스 검색.** (`src/components/Column.tsx`)

- **측정**: DOM 카드 수 5,000 → **~54개** (보이는 것 + overscan 10). 연속 스크롤 프레임
  평균 **4.1ms**, 최악 5.5ms (60fps 예산 16.7ms 의 1/4). overscan 10 은 Jira 가 같은 문제를
  풀 때 쓴 버퍼 값과 동일 (docs/enterprise-case-studies.md 5절 — Jira 는 이 방식으로 대형
  보드 마운트 시간 ~80% 감소를 보고).
- **포기한 것 ①: 카드 가변 높이.** 제목을 한 줄 말줄임으로 고정해 카드 높이를 75px 상수로
  만들었다. 덕분에 높이 실측(ResizeObserver) 없이 가장 단순한 가상화로 충분하다. 여러 줄
  카드가 필요해지면 measureElement 실측으로 전환하는 경로를 코드 주석에 남겼다.
- **포기한 것 ②: 컬럼 내 순서.** 데이터에 order 필드가 없어 순서 변경은 API 차원에서 지원
  불가 — status 이동만 지원한다 (과제 범위와 일치).
- 검색은 입력창은 즉시 반응하되 필터 계산만 250ms 디바운스. 5,000개 filter 자체는 O(n)이라
  싸지만, 타이핑마다 가상화 리스트가 다시 그려지는 것을 막는 목적이 크다.
- 드롭 이벤트는 카드가 아닌 컬럼에서 수신 — 드래그 중 원본 카드가 가상화로 언마운트돼도
  드롭이 유실되지 않는다 (Atlassian 가상화+DnD 가이드가 경고하는 함정).

## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)
- **네트워크 완전 단절** 시 동작을 어떻게 정의했나요? 왜?
- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?
- 실패한 쓰기의 **재시도**: 자동 vs 수동 — 무엇을 택했고 이유는?

**네트워크 단절 — 선발행(fail-fast) + 자동 복구.** 모든 뮤테이션은 요청 전에
`navigator.onLine` 을 확인한다. 오프라인이면 요청·재시도·낙관 반영 없이 **즉시** "네트워크
연결을 확인해주세요"를 알리고 의도를 큐에 보류하며, `online` 이벤트에 큐를 자동 재전송한다.
이유: 단절 상태에서의 재시도는 "다음엔 성공할 수도 있다"는 전제가 없는 낭비(건당 ~4초)이고,
전송 시도조차 없는 변경을 화면에 남기면 거짓 상태가 된다. 오프라인 삭제도 카드를 화면에
유지한다 — 아직 안 지워진 게 진실이므로. (PD-7. DEV 패널의 '네트워크 끊김'으로 재현 가능)

**409 충돌 — 서버 상태 채택 + 통보.** 서버가 돌려준 최신 상태(`payload.current`)로 화면을
맞추고 "다른 곳에서 먼저 수정되었습니다"를 알린다. 재시도하지 않는다(같은 version 은 영원히
409). **내 변경을 최신 version 으로 강제 재전송하는 대안은 상대방의 변경을 말없이 덮어쓰므로
기각** — 충돌 상대가 보이는 실시간 협업(Figma 식 LWW)과 달리, 이 환경에선 충돌을 드러내는
쪽이 정직하다. 참고로 카드별 직렬화 덕에 자기 자신과의 409 는 발생하지 않아, 남는 경로는
다른 탭(=다른 사용자)뿐이다. (PD-6)

**재시도 — 하이브리드(자동 3회 → 수동 일괄).** 일시 오류는 백오프로 3회 자동 재시도해
사용자 모르게 흡수하고(잔존 ~0.05%), 소진하면 롤백 + 실패 큐에 누적 + 알림에서 일괄
재시도/요청 취소를 선택하게 한다. 자동만 쓰면 영구 오류에서 무한 루프, 수동만 쓰면 15%
확률의 일시 오류마다 사용자를 방해한다. Trello 도 같은 구조(상태코드로 일시/영구 분류,
상한 있는 지수 백오프)를 쓴다. (PD-3, 사례 연구 4절)

## 6. 그 외 트레이드오프 / 남은 한계
- 시간 제약으로 포기한 것, 알고 있는 버그·한계, 더 있었다면 했을 것.

**작업 범위**: 이동·글쓰기(생성/수정)·삭제 × (장애 상태·네트워크 미연결)을 이 과제에서 풀
문제로 한정했다. 아래는 몰라서가 아니라 범위 판단으로 뺀 것들이며, 전환 조건까지 명시한다.

- **실패 큐 비영속(메모리)**: 탭을 닫으면 미전송 변경 의도가 유실된다. Linear 처럼 IndexedDB
  영속화로 풀 수 있지만 과제 범위(P2 밖)를 넘고, "앱 상태 localStorage 저장 금지" 규칙과의
  경계도 모호해진다.
- **백오프 지터 없음**: 지터는 수천 클라이언트의 동시 재시도로부터 서버를 지키는 장치다.
  단일 사용자 + mock 서버에선 재시도 폭풍이 성립하지 않는다. 실서비스 전환 시 Full Jitter
  추가가 표준 (AWS Architecture Blog).
- **멱등성 키 없음**: 이 mock 의 500은 "적용 전 거부"로 정의되어 재시도가 중복을 만들지
  않는다. 실서버라면 POST 에 Stripe 식 Idempotency-Key 가 필수다.
- **다중 탭 동기화·키보드 접근성·태그/다중 필터 미구현** (P2): 핵심(P1)의 완성도에 시간을
  집중했다. 네이티브 `<dialog>` 의 기본 포커스 트랩 정도만 확보된 상태.
- **알고 있는 한계**: 409 로 서버 상태를 채택할 때 내가 하려던 변경은 버려진다(재적용하려면
  다시 조작). 충돌 diff 를 보여주고 병합을 돕는 UX 는 발생 빈도 대비 과설계로 판단했다.

버그를 포함한 상세 결정 과정은 [docs/policy-decisions.md](docs/policy-decisions.md) (PD-1~10 +
발견·수정 버그 3건), 업계 사례 대조는 [docs/enterprise-case-studies.md](docs/enterprise-case-studies.md) 참고.
