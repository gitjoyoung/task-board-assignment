## 0. 설계 축

이 과제의 본질을 "서버는 느리고(200~800ms) 가끔 실패하는데(15%), 사용자에게는 즉각적이고 정직한 UI를 보여주는 것"으로 정의했다. 아래 모든 결정은 이 축(즉각성 ↔ 정합성)의 트레이드오프다.

## 1. 상태 구조
- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?
- 이렇게 잡은 이유와, 고려했다가 버린 대안은?

**결정: TanStack Query 도입. 서버 상태는 `['tasks']` 쿼리 캐시 하나로 관리하고, 낙관적 상태는 같은 캐시에 즉시 반영한다.**

- **invalidate → 재조회 패턴을 버렸다.** 이 API는 단건 조회가 없고 `GET /api/tasks`가 5,000개 전체를 반환한다. 교과서대로 뮤테이션마다 invalidate하면 카드 한 장 옮길 때마다 5,000개를 200~800ms 걸려 다시 받고, 그 GET도 가끔 500이 난다. 대신 **뮤테이션 응답(서버가 최신 task를 돌려줌)으로 캐시를 직접 갱신**한다. GET은 최초 1회 + 사용자가 명시적으로 재시도할 때만.
- 그래서 `staleTime: Infinity`, `refetchOnWindowFocus: false`. 자동 재조회가 낙관적 상태를 덮어쓰는 사고도 함께 차단된다.
- 초기 로드는 `retry: 2` — 명세상 GET 실패는 "드물게 500"이므로 조용한 재시도로 흡수하고, 그래도 실패하면 재시도 버튼이 있는 에러 UI를 보여준다 (`src/Board.tsx`).
- **버린 대안: `useReducer` 직접 구현.** 의존성 0이라는 장점이 있지만, 요청 취소·중복 요청 방지·스냅샷 롤백을 전부 손으로 짜야 하고 그 코드가 검증되지 않은 코드다. Query는 이 패턴이 공식 문서에 정형화되어 있다. 단, 같은 카드 연속 이동의 경쟁 상태는 Query가 자동으로 해결해주지 않으므로 3번에서 직접 보완한다.
- 부수 효과: React 18 StrictMode의 effect 이중 실행으로 초기 GET이 2번 나가던 문제도 Query의 요청 중복 제거로 해소.

## 2. 낙관적 업데이트 & 롤백
- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?
- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?

## 3. 경쟁 상태(race condition)
- 같은 카드를 빠르게 연속 이동할 때 최종 상태가 어긋나지 않도록 무엇을 했나요?
  (요청 취소 / 버전·순번 비교 / last-write-wins 등)

## 4. 대량 데이터 성능 (5,000개)
- 검색/필터/드래그를 부드럽게 만들기 위해 무엇을 선택했고 **무엇을 포기**했나요?
- 가능하면 측정 근거(예: 최적화 전/후 체감·프로파일)를 남겨 주세요.

## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)

- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?
