# 설계 결정 (DECISIONS)

> 이 문서는 평가에서 **20%**를 차지합니다. "무엇을 했는지"보다 **"왜 그렇게 했는지"**를 봅니다.
> 아래 항목을 채워 주세요. 코드로 답이 되는 부분은 파일 경로를 함께 적어도 좋습니다.

## 0. 설계 축

과제를 읽고 이 문제의 본질을 이렇게 이해했어요. 서버는 느리고(200~800ms) 가끔 실패하는데(15%), 사용자에게는 즉각적이면서도 거짓 없는 화면을 보여줘야 한다는 것. 아래 결정들은 전부 이 두 가치(즉각성, 정합성) 사이에서 고민한 결과예요.

## 1. 상태 구조

- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?

  저장소를 나누지는 않았어요. 화면이 읽는 상태는 TanStack Query의 `['tasks']` 캐시 하나예요.

  대신 역할을 나눴어요. 캐시는 지금 화면에 보여줄 값이고, 서버가 마지막으로 확정해 준 값(baseline)은 전송이 진행 중인 카드에 한해 `src/lib/taskMover.ts`가 카드 단위로 따로 들고 있어요. 실패하면 여기로 되돌리고, 다음 요청의 version도 여기서 꺼내요.

  캐시 키는 `src/hooks/useTasksQuery.ts`가, 캐시에 쓰는 일은 `src/hooks/useTaskSync.ts`가 각각 혼자 맡게 해서, 상태가 흩어지지 않도록 했어요.

- 이렇게 잡은 이유와, 고려했다가 버린 대안은?

  사용자에게 보여줄 답은 결국 한 벌이라는 생각이 출발점이었어요. 서버 사본과 낙관 오버레이를 두 벌로 두면 렌더마다 병합해야 하고, 둘이 어긋났을 때 어느 쪽이 맞는지 추적하기 어려워질 것 같았어요. 그래서 저장소는 하나로 두고, 롤백에 필요한 최소 정보만 따로 보관하는 쪽을 골랐어요.

  버린 대안 1. invalidate 후 재조회 (Query 공식 문서의 기본 패턴)
  처음엔 이 방식대로 하려고 했는데, 비용을 재 보고 생각을 바꿨어요. 이 API는 단건 조회가 없어서 재조회 1회 = 5,000건 전체 = 1,087KB예요(실측). 반면 PATCH 응답은 236바이트고요. 즉 invalidate 방식이면 카드 한 장 옮길 때마다 우리 방식의 약 4,700배 데이터를 200~800ms에 걸쳐 다시 받는 셈이에요. 게다가 그 재조회 응답이 진행 중인 다른 카드의 낙관 상태까지 덮어버려요. 그래서 뮤테이션 응답으로 캐시를 직접 갱신하고, `staleTime: Infinity`와 `refetchOnWindowFocus: false`로 자동 재조회를 막았어요. GET은 최초 1회뿐이고, 실패하면 조용히 2회 재시도한 뒤 재시도 버튼이 있는 에러 화면을 보여줘요.

  버린 대안 2. useReducer로 직접 구현
  의존성이 없다는 점이 매력적이었지만, 요청 중복 제거나 로딩/에러 추적을 전부 제가 손으로 짜야 하고, 그 코드는 검증된 적이 없는 코드라는 게 걸렸어요. 다만 Query를 쓴다고 끝나는 건 아니었어요. 같은 카드 연속 이동의 경쟁 상태는 Query가 해결해 주지 않아서, 그 부분은 직접 만들었어요. 방법은 아래 "3. 경쟁 상태" 항목에 적었어요.

## 2. 낙관적 업데이트 & 롤백

- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?

  네 동작 모두 캐시에 즉시 반영하고 백그라운드로 전송해요. 자동 재시도(0.3초 백오프 1회)까지 실패하면 그 카드만 되돌려요. 이동과 수정은 baseline으로, 생성은 임시 카드 제거로, 삭제는 스냅샷 복원으로요.

  되돌리는 시점에 인위적 대기는 없어요. 실패 응답이 도착하면 그 즉시 롤백해요. 다만 1차 실패는 "실패 확정"이 아니라 "아직 모름"으로 취급해서 한 번 더 보내 보기 때문에, 확정까지는 1차 응답(0.2~0.8초) + 백오프(0.3초) + 2차 응답(0.2~0.8초) = 0.7~1.9초가 걸려요. 실측으로는 814ms였어요. 이 판단 유예 최대 1.1초를 지불하고 사용자가 볼 알림을 15%에서 2.25%로 줄인 거예요.

  되돌리더라도 변경 의도는 버리지 않았어요. 실패 큐에 남겨서 알림에 항목별로 보여주고(무엇을 하다 실패했는지), 사용자가 일괄 재시도하거나 요청 취소로 폐기할 수 있게 했어요. 구현은 `src/lib/taskMover.ts`에 있어요.

- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?

  스냅샷(baseline) 복원을 골랐어요. 이유는 두 가지였어요.

  첫째, 재조회는 5,000건 GET이라 롤백 한 번에 200~800ms와 실패 가능성을 또 지불해야 해요.
  둘째, 재조회 응답은 캐시 전체를 덮어서, 그 순간 진행 중이던 다른 카드들의 낙관 상태까지 지워버려요. 스냅샷 복원은 실패한 카드만 정확히 되돌릴 수 있었어요.

  스냅샷의 기준을 "동작 직전"이 아니라 "서버가 마지막으로 확정한 상태"로 둔 것도 고민한 부분이에요. 연속 이동 중 두 번째가 실패했을 때 맨 처음으로 돌아가면 성공한 첫 이동까지 잃어버리게 되니까, 첫 이동이 확정된 지점으로 돌아가도록 했고 테스트로 고정해 뒀어요.

## 3. 경쟁 상태(race condition)

- 같은 카드를 빠르게 연속 이동할 때 최종 상태가 어긋나지 않도록 무엇을 했나요?
  (요청 취소 / 버전·순번 비교 / last-write-wins 등)

  보기 중에서 고르면 버전 비교에 가까운데, 정확히는 세 가지를 조합했어요. 구현과 테스트는 `src/lib/taskMover.ts`에 있어요.

  카드별 직렬화예요. 카드당 동시 요청을 1개로 제한하고, 진행 중에 들어온 변경은 병합해 두었다가 이전 응답이 도착하면 그 응답의 version으로 이어서 보내요. 이렇게 한 이유는 숫자로 명확한데, 카드를 3번 연속 이동할 때 병렬로 보내면 3건 중 2건이 409로 실패해요(전부 version 1을 들고 나가니까요). 직렬화하면 409가 0건이에요.

  연타 병합이에요. A에서 B, C를 거쳐 D로 빠르게 옮기면 중간 상태는 보내지 않고 최종 목표만 보내요. 위의 3연속 이동이 전송 2건(첫 요청 + 병합된 최종)으로 끝나요. 사용자가 원한 건 결국 D니까요.

  늦은 응답 무시예요. 첫 요청의 응답이 도착했을 때 화면에 더 새로운 낙관 상태가 있으면, status는 그대로 두고 서버 메타(version 등)만 받아들여요. 나중에 찾아보니 Figma가 화면 깜빡임을 막는 데 같은 원리를 쓰고 있었어요(docs/enterprise-case-studies.md 2절). 방향이 틀리지 않았다는 확인이 됐어요.

  보기의 나머지를 안 쓴 이유도 있어요. 요청 취소(fetch abort)는 이 서버에서는 오히려 위험해 보였어요. 서버가 요청을 받는 순간 처리해 버리니까, 취소해도 서버 상태는 바뀌고 저만 응답을 못 받는 상황이 돼요. last-write-wins 강제는 상대방의 변경을 말없이 덮어쓰게 돼서 버렸어요. 자세한 이유는 아래 "5. 정답이 없던 결정들"의 409 항목에 적었어요. 삭제만 예외로 직렬화하지 않았는데, DELETE는 version을 안 쓰고 멱등이라 병렬로 보내도 안전하다고 판단했어요.

## 4. 대량 데이터 성능 (5,000개)

- 검색/필터/드래그를 부드럽게 만들기 위해 무엇을 선택했고 **무엇을 포기**했나요?

  선택한 건 두 가지예요. `@tanstack/react-virtual` 고정 높이 가상화(`src/components/Column.tsx`), 그리고 검색 입력 250ms 디바운스요. 드롭 이벤트는 카드가 아니라 컬럼에서 받도록 했는데, 드래그 중에 원본 카드가 가상화 때문에 화면에서 내려가도 드롭이 유실되지 않게 하기 위해서였어요.

  포기한 것 1. 카드 가변 높이
  제목을 한 줄 말줄임으로 고정해서 카드 높이를 75px 상수로 만들었어요. 그 덕에 높이 실측 없이 가장 단순한 가상화로 충분해졌어요. 여러 줄 카드가 필요해지면 실측(measureElement)으로 전환하면 되고, 그 경로는 코드 주석에 적어 뒀어요.

  포기한 것 2. 컬럼 내 순서 변경
  데이터에 order 필드가 없어서 API 차원에서 불가능했어요. status 이동만 지원해요. 대신 표시 규칙을 하나 정했어요 — 이동한 카드는 대상 컬럼 맨 위에 보여요. 자기 배열 위치로 들어가면 가상화 창 밖(수백 번째 줄)에 떨어져서 이동이 반영 안 된 것처럼 보이는 문제가 있었거든요. 순서는 서버 데이터가 아니라 세션 내 표시 규칙이라, 새로고침하면 서버 순서로 돌아가요.

- 가능하면 측정 근거(예: 최적화 전/후 체감·프로파일)를 남겨 주세요.

  DOM에 실리는 카드가 5,000개에서 약 54개로, 98.9% 줄었어요(보이는 것 + 버퍼 10개인데, 나중에 보니 Jira가 같은 문제에 쓴 버퍼 값과 같았어요). 연속 스크롤 프레임은 평균 4.1ms, 최악 5.5ms로 60fps 예산(16.7ms)의 25~33% 수준이에요. 검색 디바운스는 filter 연산 자체(O(n)이라 저렴해요)보다, 타이핑마다 가상 리스트가 다시 그려지는 걸 막으려고 넣었어요.

## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)

- **네트워크 완전 단절** 시 동작을 어떻게 정의했나요? 왜?

  "시도하지 않고 즉시 알리되, 화면은 즉시 반영한다"로 정했어요.

  모든 뮤테이션이 요청 전에 `navigator.onLine`을 확인해요. 오프라인이면 요청 없이 바로 "네트워크 연결을 확인해주세요"를 알리고 의도를 큐에 보류해요. 이동과 수정은 화면에 먼저 반영해 두고, 연결이 돌아오면(online 이벤트) 큐를 자동으로 재전송해요. 요청 취소를 누르면 화면도 서버 상태로 되돌려요.

  숫자로 비교하면 이래요. 오프라인 드롭을 일반 실패 경로로 태우면 확정까지 0.7~1.9초 + 무의미한 요청 2건이고, 선발행은 알림까지 150ms 이내(실측)에 요청 0건이에요. 단절 상태에서는 "다음엔 성공할 수도 있다"는 재시도의 전제 자체가 없으니까요.

  화면 반영은 솔직히 적으면 뒤집은 결정이에요. 처음엔 전송 시도조차 없는 변경을 화면에 남기면 거짓 상태라고 생각해서 반영하지 않았는데, 직접 써 보니 드래그했는데 화면이 아무 반응이 없는 게 더 큰 혼란이더라고요. 그래서 화면은 반영하되 알림으로 "아직 저장 안 됨"을 바로 알리고, 요청 취소가 화면까지 되돌리는 지금 방식으로 바꿨어요(docs/policy-decisions.md PD-7). DEV 패널의 네트워크 끊김 토글로 재현해 볼 수 있어요.

- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?

  서버가 돌려준 최신 상태로 화면을 맞추고 "다른 곳에서 먼저 수정되었습니다"를 보여줘요. 재시도는 하지 않아요.

  같은 version을 다시 보내면 영원히 409라서 재시도가 의미가 없었어요. 최신 version으로 강제 재전송하는 방법도 생각해 봤는데, 그건 상대방(다른 탭)의 변경을 말없이 덮어쓰는 거라서요. 상대 커서가 보이는 실시간 협업 도구라면 몰라도, 이 환경에서는 충돌이 났다는 사실을 사용자에게 보여주는 쪽이 맞다고 판단했어요. 참고로 카드별 직렬화 덕분에 자기 자신과 충돌할 일은 없어서, 남는 409 경로는 다른 탭뿐이에요.

- 실패한 쓰기의 **재시도**: 자동 vs 수동 — 무엇을 택했고 이유는?

  둘 다 써요. 일시 오류는 0.3초 뒤 1회 자동 재시도로 조용히 흡수하고(흡수율 97.75%), 그래도 실패하면 롤백하고 알림에서 수동으로 일괄 재시도하거나 요청 취소할 수 있게 했어요. 재시도를 눌러도 알림은 닫히지 않아요 — 재시도도 실패할 수 있으니까, 성공이 확정된 항목부터 목록에서 빠지고 전부 해소돼야 닫혀요.

  자동만 쓰면 영구 오류에서 무한 루프가 되고, 수동만 쓰면 15% 확률의 일시 오류마다 사용자를 귀찮게 해요. 자동 횟수는 처음에 3회로 잡았다가 1회로 줄였는데, 이것도 써 보고 바꾼 결정이에요. 3회는 실패 확정까지 4.5초쯤 걸려서, 카드가 한참 뒤에 혼자 되돌아오는 게 오히려 혼란스러웠어요. 즉각 인지가 흡수율보다 중요하다고 결론 냈어요(PD-3). 나중에 Trello 기술 블로그에서 같은 구조(상태코드로 일시/영구 분류, 상한 있는 백오프)를 확인하고 참고가 됐어요.

## 6. 그 외 트레이드오프 / 남은 한계

- 시간 제약으로 포기한 것, 알고 있는 버그·한계, 더 있었다면 했을 것.

  포기한 것들이에요. 몰라서 뺀 게 아니라는 걸 남기고 싶어서, 언제 다시 넣으면 될지도 같이 적어요.

  - 실패 큐가 메모리에만 있어요. 탭을 닫으면 미전송 의도가 사라져요. Linear처럼 IndexedDB로 영속화할 수 있지만 P2 밖이고, 앱 상태의 localStorage 저장 금지 규칙과 경계가 모호해질 것 같았어요.
  - 백오프에 지터가 없어요. 지터는 수천 클라이언트가 동시에 재시도할 때 서버를 지키는 장치라, 단일 사용자와 mock 서버 조합에서는 성립하지 않는 문제라고 봤어요. 실서비스로 가면 Full Jitter가 표준이에요(AWS Architecture Blog).
  - 멱등성 키가 없어요. 이 mock의 500은 적용 전 거부라서 재시도가 중복을 만들지 않지만, 실서버라면 POST에 Stripe식 Idempotency-Key가 필요해요.
  - 키보드 접근성, 태그/다중 필터(P2)는 P1 완성도에 시간을 쓰느라 뺐어요.

  알고 있는 한계예요. 409로 서버 상태를 받아들이면 제가 하려던 변경은 버려져요. 다시 하려면 다시 조작해야 해요. 충돌 diff를 보여주고 병합을 돕는 UX까지는 발생 빈도 대비 과하다고 봤는데, 더 나은 답이 있을 수 있다고 생각해요.

  다중 탭 동기화(P2)는 구현했어요. 서버가 확정한 변경만 BroadcastChannel로 다른 탭에 방송하고, 받는 쪽은 version이 더 새것일 때만 반영해요. 방송 내용물이 이미 손에 있는 서버 응답이라 추가 요청은 0건이에요. 실서버 기준으로 생각해 보면 다중 탭 문제는 "내 다른 탭"(브라우저 안에서 해결 가능, 이번 구현)과 "다른 사용자"(서버 푸시가 필요, 이 환경에선 재현 불가)로 나뉘는데, 전자가 과제 범위라고 판단했어요. 상세는 docs/policy-decisions.md PD-11에 있어요.

  더 있었다면 하고 싶었던 것들이에요. 실패 큐 영속화와 재접속 시 자동 전송(Linear 패턴), 키보드 드래그(dnd-kit 재검토), 그리고 실서버 전환 시 WebSocket 푸시로 "다른 사용자" 신선도까지 확장하는 것요.
