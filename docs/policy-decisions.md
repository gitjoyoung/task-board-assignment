# 정책 결정 기록 (Policy Decision Records)

구현 중 내린 정책 결정들의 상세 기록이다. 각 항목은 **결정 / 근거 / 기각한 대안 / 검증** 순서로 적는다.
루트의 DECISIONS.md 는 이 기록의 요약본이며, 업계 사례 대조는 [enterprise-case-studies.md](./enterprise-case-studies.md) 참고.

## PD-1. 서버 상태는 뮤테이션 응답으로 직접 갱신한다 (invalidate 금지)

**결정**: TanStack Query 를 쓰되 교과서 패턴(뮤테이션 후 invalidate → 재조회)을 버리고,
PATCH/POST 응답이 돌려주는 최신 task 로 캐시를 직접 갱신한다. `staleTime: Infinity`,
`refetchOnWindowFocus: false`.

**근거**: 이 API 는 단건 조회가 없고 GET 이 5,000개 전체를 반환한다. invalidate 를 쓰면
카드 한 장 옮길 때마다 5,000개를 200~800ms 걸려 다시 받고, 그 GET 도 가끔 500이 난다.
자동 재조회는 낙관적 상태를 옛 데이터로 덮어쓰는 사고 경로이기도 하다.

**기각한 대안**: invalidate 패턴(재조회 폭풍), useReducer 직접 구현(요청 중복 제거·상태 추적을
전부 손으로 재작성).

**검증**: 연속 이동 시 네트워크 로그에 GET 재발행 없음.

## PD-2. 실패 확정 시 즉시 롤백한다 (보류형 기각)

**결정**: 자동 재시도까지 소진해 실패가 확정되면 그 즉시 baseline(마지막 서버 확정 상태)으로
롤백하고 알림을 띄운다. 재시도 알림이 떠 있는 동안 화면은 이미 롤백된 상태다.

**근거**: 과제 명세 P1 이 "실패하면 이전 상태로 정확히 되돌리고 알림"을 요구한다. 설계
관점에서도, 낙관적 상태를 유지한 채 사용자 결정을 기다리면 "진행 중인 요청도 없이 서버와
다른 화면"이 무한정 유지되는 거짓 상태가 된다. 낙관은 성공을 낙관할 근거(진행 중인 시도)가
있을 때만 정당하다. 자동 재시도가 도는 동안(최대 ~4초)은 낙관 상태가 유지되므로 대부분의
일시 오류에서 사용자는 롤백을 보지 못한다.

**기각한 대안**:
- 실패 후에도 낙관 상태 유지 + 사용자 결정까지 보류(거짓 상태 장기화, 명세 위반)
- **다이얼로그 선행·조건부 롤백**(실패 시 모달로 재시도/취소를 묻고 취소일 때만 롤백):
  모달이라 거짓 상태 위 조작은 차단되지만, ① 제출 체크리스트의 "실패율을 높여 롤백 동작
  확인" 시나리오에서 롤백이 보이지 않고 ② 다건 실패에서 모달이 건별로 쌓여 UX 가 무너지며
  ③ 현재 방식도 의도를 실패 큐에 보존하므로 "사용자 결정으로 최종 상태가 정해진다"는 목적은
  이미 달성된다 — 차이는 대기 중 화면이 진실(현재)이냐 희망(대안)이냐 뿐.

**검증**: taskMover.test.ts "실패하면 해당 카드만 이동 전 상태로 롤백하고 사용자에게 알린다".

## PD-3. 재시도는 하이브리드 — 자동 3회 후 수동 일괄

**결정**: 일시 오류(500)는 백오프(0.3/0.6/1.2초)로 최대 3회 자동 재시도해 조용히 흡수한다.
소진하면 롤백 + 실패 큐 누적 + 알림. 알림의 재시도 버튼은 큐 **전체**를 일괄 재시도한다.

**근거**: 실패율 15% 기준 자동 3회로 실패의 99.95%가 흡수돼(0.15⁴) 사용자는 진짜 장애일 때만
알림을 본다. 여러 건이 연속 실패했을 때 마지막 1건만 재시도되면 나머지는 기회를 잃는다 —
큐 단위 일괄 재시도가 맞다.

**기각한 대안**: 무한 자동 재시도(영구 오류에서 무한 루프), 수동만(일시 오류마다 사용자 방해),
건별 재시도 버튼(다건 실패 시 앞선 실패가 유실).

**검증**: "일시 오류는 자동 재시도로 흡수된다", "여러 카드가 실패하면 전부 누적되고 retryFailed 는
전부 다시 시도한다".

## PD-4. 실패 알림은 2-선택지 — 재시도 / 요청 취소

**결정**: 실패 알림의 선택지는 둘뿐이다.
- **재시도**: 실패 큐 전체 재전송
- **요청 취소**: 큐 폐기(변경 의도 포기). 화면은 이미 롤백돼 있으므로 의도만 버린다

알림은 결정 전까지 유지되지만 비차단(토스트)이라 작업을 막지 않는다. 실패 건수가 0인
정보성 알림(409 등)은 버튼 없이 4초 후 자동 소멸한다. 네트워크 복구로 큐가 자동
재전송되면 알림도 함께 닫힌다(재실패 시 다시 뜸).

**근거**: 처음엔 "닫기(✕) = 알림만 숨기고 큐 유지"라는 세 번째 선택지가 있었으나 제거했다.
숨긴 큐를 다시 볼 UI 가 없어서, 닫기를 누르는 순간 대기 의도가 다음 실패 전까지 보이지 않는
유령 상태로 남기 때문이다. 두 선택지면 모든 실패가 명시적으로 해소되고 유령 상태가 원천
제거된다. "결정 미루기"는 버튼이 아니라 토스트의 비차단성이 이미 제공한다.

**기각한 대안**: 닫기(숨김) 버튼 유지(유령 큐 상태), 자동 소멸(실패를 놓침).

**검증**: "discardFailed 는 실패 큐를 폐기한다 — 이후 retryFailed 가 아무것도 재생하지 않는다".

## PD-5. 실패 알림 중 보드 조작을 차단하지 않는다

**결정**: 재시도 알림이 떠 있어도 보드는 정상 조작 가능하다.

**근거**: 알림 시점의 화면은 이미 서버 확정 상태(정직한 상태)라 어떤 조작도 안전하다.
실패한 카드를 손으로 다시 움직이면 큐의 옛 의도가 대체되므로(아래 PD-8) 이중 재생 사고도
구조적으로 불가능하다. 카드 1장의 실패로 나머지 4,999장을 잠그는 건 카드 단위 격리 설계를
스스로 무너뜨린다. 차단이 정당한 경우는 삭제 확인처럼 그 결정 자체를 받아야 하는 모달뿐이다.

**기각한 대안**: 알림 중 전체 차단(즉각성 상실 — 15% 실패율에서 빈번한 정지).

## PD-6. 409 충돌은 재시도하지 않고 서버 상태를 채택한다

**결정**: 409 수신 시 서버가 돌려준 최신 상태(`payload.current`)로 화면을 맞추고 알린다.
자동 재시도 대상에서 제외한다.

**근거**: 같은 version 으로 재전송하면 영원히 409다(영구 오류). 최신 version 으로 강제
재전송하는 대안은 상대방(다른 탭)의 변경을 말없이 덮어쓴다 — "정직한 UI" 원칙 위반.
참고로 자기 자신과의 409 는 카드별 직렬화(다음 요청이 직전 응답의 version 사용)로 원천
차단했으므로, 남는 409 경로는 동시 편집자뿐이다.

**기각한 대안**: 최신 version 재전송(last-writer-wins 강제 — 상대 변경 무단 덮어쓰기),
diff 병합 UI(과제 범위 대비 과설계).

**검증**: "409 충돌 시 서버 최신 상태를 채택하고 알린다", "409 는 재시도하지 않는다".

## PD-7. 네트워크 단절은 선발행 — 시도하지 않고 즉시 알린다

**결정**: 모든 뮤테이션은 요청 전에 `navigator.onLine` 을 확인한다. 오프라인이면 요청·자동
재시도·낙관 반영 없이 즉시 "네트워크 연결을 확인해주세요" 알림 + 실패 큐 보류. `online`
이벤트 수신 시 큐를 자동 재전송한다.

**근거**: 단절 상태의 재시도는 "다음 시도는 성공할 수도 있다"는 전제가 없는 낭비다(건당
최대 ~4초 소모). 오프라인 중 낙관 반영을 하지 않는 이유: 전송 시도조차 없는 변경을 화면에
남기면 PD-2 가 금지한 거짓 상태가 된다. 삭제도 카드를 화면에 유지한다(아직 안 지워진 게 진실).

**기각한 대안**: 오프라인에도 일반 실패 경로로 처리(무의미한 재시도 낭비), 오프라인 낙관 반영
+ 로컬 영속 큐(Linear 식 local-first — 올바르지만 과제 범위 밖, 탭 수명 내 메모리 큐로 충분).

**검증**: "오프라인이면 요청·재시도 없이 즉시 알리고 큐에 쌓는다", "온라인 복귀 후 retryFailed 가
대기 큐를 그대로 전송한다". DEV 패널의 '네트워크 끊김'은 navigator.onLine 자체를 오버라이드해
실제 경로 그대로 검증한다.

## PD-8. 실패 큐의 키 불변식 — 반복은 병합되고, 수동 조작이 우선한다

**결정**:
- 이동/수정/삭제 의도는 **카드 id** 를 키로: 같은 카드의 반복 실패는 1건으로 병합
- 생성 의도는 **내용 키**(`create:` + 입력 JSON): 같은 요청 반복도 1건으로 병합
- 실패한 카드를 손으로 다시 조작하면 큐의 옛 의도가 대체된다
- 같은 컬럼 드롭(no-op)은 요청도 안 보내고 큐도 건드리지 않는다

**근거**: 생성 의도를 시도별 임시 uuid 로 키잉했더니 같은 요청 반복이 무한 누적되는 버그가
있었다(발견: 사용자, 원인: 위임 코드의 키 체계 미감사). 큐는 "사용자 의도"의 집합이므로
동일 의도는 하나여야 하고, 더 새로운 수동 조작이 항상 옛 의도를 이긴다.

**검증**: "동일 내용의 생성을 반복해도 실패 큐에는 한 건으로 합쳐진다", "실패한 카드를 손으로
다시 이동하면 실패 큐에서 빠진다", "같은 컬럼 드롭(no-op)은 큐에 있던 실패 의도를 지우지 않는다".

## PD-9. 삭제는 직렬화하지 않는다

**결정**: DELETE 는 진행 중인 수정과 직렬화하지 않고 즉시 병렬로 보낸다. 대신 삭제된 카드
id 를 `removed` 로 표시해, 늦게 도착하는 수정 응답이 카드를 되살리지 못하게 가드한다.

**근거**: DELETE 는 version 을 쓰지 않아 409 여지가 없고 멱등이다(없는 id 도 204). 직렬화는
version 경쟁이 있는 PATCH 에만 필요한 비용이다.

**검증**: taskMover 삭제 테스트 일체. (선택 근거는 코드 주석에도 기록)

## PD-10. 지터·멱등성 키·큐 영속화는 의도적으로 생략한다

**결정**: 재시도 백오프에 지터를 넣지 않고, 생성 요청에 멱등성 키를 붙이지 않으며,
실패 큐를 localStorage 등에 영속화하지 않는다.

**근거**:
- **지터**: 수천 클라이언트의 동시 재시도로부터 서버를 지키는 장치다. 단일 사용자 + mock
  서버에선 재시도 폭풍이 성립하지 않는다. 실서비스 전환 시 Full Jitter 추가가 표준
  (AWS Architecture Blog 참고).
- **멱등성 키**: mock API 가 지원하지 않고, 이 mock 의 500은 "적용 전 거부"로 정의되어
  재시도가 중복을 만들지 않는다. 실서버라면 POST 에 Stripe 식 Idempotency-Key 필수.
- **큐 영속화**: 탭을 닫으면 미전송 변경이 유실된다. Linear 는 IndexedDB 로 해결하지만
  과제 범위(P2 밖)를 넘고, 앱 상태의 localStorage 저장 금지 규칙과의 경계도 모호해진다.

이 셋은 몰라서가 아니라 범위 판단으로 뺀 것이며, 전환 조건까지 여기 명시한다.
