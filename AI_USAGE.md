# AI 도구 활용 (AI_USAGE)

## 1. 사용한 도구

- **Claude Code** (CLI 에이전트) — 설계 논의, 구현, 테스트 작성, 브라우저 검증, 기술 문서 리서치
- 구현 일부는 Claude Code 안에서 서브에이전트(Sonnet)에 맡기고, 플랜 작성과 결과물 리뷰는 상위 모델이 담당하는 식으로 비용을 조절했다

## 2. 주로 어디에 썼나

| 작업 | 활용 방식 |
|------|-----------|
| 스타터 구조 분석·결함 진단 | 코드 전체를 읽히고 "그대로 확장하면 안 되는 지점"을 보고받아 설계 출발점으로 사용 |
| 상태 관리 방식 결정 | TanStack Query vs 직접 구현 트레이드오프를 논의 — 최종 결정은 내가 (DECISIONS.md 1번) |
| 낙관적 동기화 모듈(taskMover) | 기대 동작을 먼저 합의 → 테스트로 스펙 고정(레드) → 구현(그린) TDD 순서를 강제 |
| CRUD·검색/필터 구현 | 상세 플랜(시그니처·동작 스펙·테스트 목록)을 확정한 뒤 위임, 결과물은 리뷰 후 수정 |
| 모든 기능의 실동작 검증 | 브라우저 자동화로 낙관 반영→실패→롤백→재시도 전 과정을 실제로 돌려 확인 |
| 업계 사례 리서치 | Trello/Figma/Linear/AWS/Stripe/Jira 기술 문서 조사 (docs/enterprise-case-studies.md) |

## 3. AI 답을 그대로 쓰지 않고 검증·거부·수정한 사례 (중요)

**사례 1 — AI 가 만든 실패 큐 키잉 버그를 발견·수정시킴.**
AI 가 구현한 실패 큐에서 생성(create) 의도가 시도마다 새 임시 uuid 를 키로 사용했다.
같은 생성을 반복하면 실패 건수가 1→2→…→10으로 무한 누적됐다. 쓰는 도중에 발견했는데
("이동을 안 했는데 변경 10건이 처리 안 됐다는데"), E2E 검증만으로는 못 잡는 문제였다.
코드를 정독시켜 큐 키 체계를 경로별로 감사하게 했고, 생성 의도는 내용 기반 키로 병합하도록
고친 뒤 회귀 테스트를 남겼다. 같은 감사에서 두 번째 버그(같은 컬럼 드롭이 큐 의도를 삭제하는
가드 순서 문제)도 함께 발견됐다. → docs/policy-decisions.md PD-8

**사례 2 — AI 의 초기 알림 설계가 다건 실패를 유실하는 것을 지적.**
초기 구현은 알림 상태가 단일 값이라 5건이 연속 실패하면 마지막 1건만 재시도할 수 있었다.
"내가 이동한 전부가 재시도되어야 하는데 최초 1개만 재시도되는 것 같다"고 지적해
실패 큐와 일괄 재시도 구조로 재설계시켰다. → PD-3

**사례 3 — 교과서 패턴(invalidate)을 API 구조 근거로 기각.**
TanStack Query 의 표준 패턴은 뮤테이션 후 invalidate → 재조회지만, 이 API 는 단건 조회가
없고 GET 이 5,000개 통짜라 카드 1장 이동마다 대형 재조회가 발생한다. 공식 문서의 권장을
그대로 따르지 않고 "뮤테이션 응답으로 캐시 직접 갱신"으로 바꿨다. → PD-1

**사례 4 — AI 의 커밋·구현 진행 방식을 교정.**
AI 가 내 아이디어(자동 재시도)를 명세 대조 없이 바로 구현하기 시작해서 중단시키고,
"가설은 README 의 P1/P2/판단 항목과 대조해 확인받고 구현"하는 절차를 규칙으로 박았다.
테스트 파일 위치를 임의로 옮긴 것도 스타터 컨벤션(콜로케이션)대로 원복시켰다.

## 4. 스스로 판단해서 결정한 부분

- **작업 범위 확정**: 이동·글쓰기·삭제 × (장애 상태·네트워크 미연결)을 이 과제에서 풀 문제로
  한정하고 나머지(다중 탭, 키보드 접근성 등)는 근거를 남기고 제외
- **오프라인 정책**: "요청 시점에 연결을 선확인하고, 오프라인이면 재시도 없이 즉시 알린다"는
  선발행 방식은 내 결정 — AI 는 선택지(감지+자동복구 / 서킷 브레이커 / 문서화만)를 정리해
  제시했고 최종 방향은 내가 골랐다 → PD-7
- **알림 UX**: 재시도/요청 취소 2-선택지로 단순화(닫기 버튼 제거), 실패 항목별 명시 표시는
  내 요구로 추가. 반대로 "실패 알림 중 보드 조작 차단"은 내가 제안했다가 AI 의 반론
  (정직한 상태 위의 조작은 안전, 카드 단위 격리 훼손)을 듣고 철회 → PD-4, PD-5
- **롤백 시점**: "다이얼로그로 물어보고 나서 롤백" 대안을 내가 제기했고, 평가 시나리오와
  다건 실패 UX 근거로 기각하는 데 합의 → PD-2

전체 정책 결정의 상세 기록은 [docs/policy-decisions.md](docs/policy-decisions.md),
업계 사례 대조는 [docs/enterprise-case-studies.md](docs/enterprise-case-studies.md) 참고.
