# AI 도구 활용 (AI_USAGE)

## 1. 사용한 도구

- **Claude Code** (CLI 에이전트) — 설계 논의, 구현, 테스트 작성, 브라우저 검증, 기술 문서 리서치에 썼어요.
- 구현 일부는 Claude Code 안에서 서브에이전트(Sonnet)에 맡기고, 플랜 작성과 결과물 리뷰는 상위 모델이 하는 식으로 비용을 조절했어요.

## 2. 주로 어디에 썼나

개발 진행 순서대로 적었어요.

| 단계 | 작업 | 사용한 도구 | 활용 방식 |
|------|------|-------------|-----------|
| 1. 요구사항 분석 | 과제 해석, 우선순위 판단 | Claude | 과제를 먼저 제가 해석하고, AI의 해석과 비교해 서로 놓친 걸 걸러냈어요. P1/P2와 평가 비중 기준으로 작업 순서를 세웠어요 |
| 2. 현황 분석 | 스타터 코드 파악, 결함 진단 | Claude | 코드 전체를 읽혀 "그대로 확장하면 안 되는 지점"(전량 재조회 구조, no-op 드롭 등)을 보고받고 교차 확인했어요 |
| 3. 설계 | 상태 구조, 동기화 정책 결정 | Claude | 갈림길(Query vs 직접 구현, 롤백 방식 등)마다 트레이드오프를 논의하고 결정은 제가 했어요 (DECISIONS 참고) |
| 4. 기술 검토 | 업계 사례 리서치 | Claude + 웹 검색/원문 조회 도구 | 검색 요약에 의존하지 않고 Trello, Figma, Linear, AWS, Stripe, Jira의 1차 기술 문서를 직접 조회시켜 설계 근거로 정리했어요 (docs/enterprise-case-studies.md) |
| 5. 구현 | taskMover, CRUD, 검색, 가상화 | Claude (일부 서브에이전트 위임) | 테스트로 스펙 고정(레드) 후 구현(그린)하는 TDD 순서를 강제했어요. 위임분은 플랜을 확정해 주고 결과물을 리뷰했어요 |
| 6. 테스트/검증 | 유닛·통합 테스트, 실동작 확인 | Claude + Chrome 브라우저 제어 MCP | 테스트 통과와 별개로, 실제 브라우저를 MCP로 조작해(드래그, 오프라인 토글, 탭 2개) 실패, 롤백, 재시도, 다중 탭 동기화를 눈으로 확인했어요. 유닛으로 못 잡는 버그를 여기서 잡았어요 (사례 4) |
| 7. 문서화 | DECISIONS, 정책 기록 | Claude | 결정 시점마다 근거를 기록하게 하고, 제출 문서는 문체를 다듬었어요 |

## 3. AI 답을 그대로 쓰지 않고 검증·거부·수정한 사례 (중요)

**사례 1. AI가 만든 실패 큐 버그를 사용 중에 발견했어요.**
AI가 구현한 실패 큐에서 생성(create) 의도가 시도마다 새 임시 uuid를 키로 쓰고 있었어요. 같은 생성을 반복하면 실패 건수가 1, 2, ... 10으로 계속 쌓이는 걸 직접 쓰다가 발견했어요. E2E 검증은 통과한 상태였는데도요. 코드를 정독시켜서 큐 키 체계를 경로별로 감사하게 했고, 생성 의도를 내용 기반 키로 병합하도록 고친 뒤 회귀 테스트를 남겼어요. 같은 감사에서 두 번째 버그(같은 컬럼 드롭이 큐 의도를 지우는 가드 순서 문제)도 나왔어요. → docs/policy-decisions.md PD-8

**사례 2. 실제 사용자처럼 써 보는 테스트로 다건 실패 유실을 찾아냈어요.**
AI의 초기 구현은 단건 실패 시나리오에서는 멀쩡했어요. 그런데 실제 사용자라면 카드 한 장만 옮기지 않죠. 서버 실패 조건을 걸어 두고 카드 여러 장을 연달아 옮겨 보니, 알림 상태가 단일 값이라 마지막 1건만 재시도되고 앞선 실패들은 조용히 사라졌어요. 이 시나리오를 근거로 실패 큐와 일괄 재시도 구조로 다시 만들게 했어요. 코드가 통과시키는 시나리오와 사용자가 실제로 만드는 시나리오의 간극을 메우는 게 사람의 역할이라고 생각해요. → PD-3

**사례 3. 표준 패턴을 이 API 구조에 맞게 커스텀했어요.**
TanStack Query의 기본 패턴은 뮤테이션 후 invalidate와 재조회인데, 이 API는 단건 조회가 없고 GET이 5,000건 통짜예요. 그대로 적용하면 카드 1장 이동마다 대형 재조회가 생기는 구조라, 뮤테이션 응답(서버가 확정한 최신 task)으로 캐시를 직접 갱신하는 방식으로 맞춤 설계했어요. 덕분에 수정 후 반영 확인이 재조회 없이 응답 그 자체로 끝나고(version 증가로 검증), 카드 1장을 옮겨도 추가 조회 0건에, 진행 중인 다른 카드의 낙관 상태도 보존돼요. 표준을 아는 것과 별개로, 표준이 전제하는 조건(단건 조회 API)이 없는 환경에서는 그에 맞게 조정할 수 있어야 한다고 배웠어요. → PD-1

**사례 4. 유닛 테스트는 통과했는데 실제 브라우저에서 안 되는 버그가 있었어요.**
다중 탭 동기화를 붙였을 때 유닛 테스트 66개가 전부 통과했는데, 실제 탭 2개를 열어 보니 동기화가 안 됐어요. React StrictMode가 mount, cleanup, 재mount를 도는 과정에서 이미 닫힌 BroadcastChannel에 다시 구독하는 생명주기 버그였어요. 채널 소유권을 effect로 옮겨 고쳤고, 유닛만 믿으면 안 된다는 걸 배웠어요. → PD-11

**사례 5. AI의 작업 방식 자체도 교정했어요.**
AI가 제 아이디어(자동 재시도)를 과제 명세와 대조하지 않고 바로 구현하기 시작해서 중단시키고, "가설은 README의 P1/P2/판단 항목과 대조해 확인받고 구현"하는 절차를 규칙으로 만들었어요. 테스트 파일 위치를 임의로 옮긴 것도 스타터 컨벤션(콜로케이션)대로 되돌리게 했어요.

## 4. 스스로 판단해서 결정한 부분

- **작업 범위 정의**: 이동, 글쓰기, 삭제에 장애 상태와 네트워크 미연결을 곱한 범위를 이 과제에서 풀 문제로 정했어요. 다중 탭 요구도 실서버 관점에서 "내 다른 탭"(브라우저 안에서 해결 가능)과 "다른 사용자"(서버 푸시 필요, 이 환경에선 재현 불가)로 나눠서 전자만 범위에 넣었어요. → PD-11
- **검증 환경을 먼저 구축**: 본 기능보다 앞서, 실패율과 지연과 네트워크 끊김을 화면에서 바로 조절하는 DEV 패널부터 만들자고 정했어요. 이 과제의 핵심이 실패 조건에서의 동작인 만큼, 모든 시나리오를 언제든 재현할 수 있는 도구가 개발 속도와 검증 신뢰도를 결정한다고 봤어요.
- **오프라인 정책**: "요청 시점에 연결을 먼저 확인하고, 오프라인이면 재시도 없이 즉시 알린다"는 선발행 방향은 제가 정했어요. AI는 선택지(감지+자동복구, 서킷 브레이커, 문서화만)를 정리해 줬고 고르는 건 제 몫이었어요. → PD-7
- **알림의 정보 설계**: 실패 알림이 "무엇을 하다 실패했는지"까지 말해줘야 한다고 봤어요. 이동은 어느 컬럼에서 어느 컬럼으로, 수정은 어떤 필드를 바꾸다가, 생성과 삭제는 어떤 카드였는지를 항목별로 표시하게 했고, 선택지는 재시도와 요청 취소 둘로 좁혀 모든 실패가 명시적으로 해소되게 했어요. → PD-4
- **정책 수치는 사용 경험으로 검증**: 재시도 횟수나 알림 문구 같은 수치와 문구는 머리로 정하고 끝내지 않고, 실패 조건을 걸어 두고 직접 써 보면서 확정했어요. → PD-3
- **구조 품질 기준**: 기능 완성 후에 관심사 분리(서버 상태 훅, 동기화 훅, 프레젠테이션 컴포넌트)와 커버리지 측정을 요구했어요. 조립 계층(Board)에는 로직이 남지 않게, 테스트는 위험이 큰 정책 계층에 가장 두껍게요.

정책 결정의 상세 기록은 [docs/policy-decisions.md](docs/policy-decisions.md), 업계 사례 조사는 [docs/enterprise-case-studies.md](docs/enterprise-case-studies.md)에 있어요.
